# Семантика фильтров Model Studio CS

Этот документ фиксирует целевое поведение фильтров спецификатора в терминах, близких к Model Studio CS.

## Поддержанные операторы

| Оператор | Назначение | Пример |
|---|---|---|
| `=` / `==` | Равно | `[PART_TYPE] = "Арматура"` |
| `<>` / `!=` | Не равно | `[BOM_INCLUDE] <> 0` |
| `>` | Больше | `[PART_WEIGHT] > 10` |
| `<` | Меньше | `sort < 99` |
| `>=` | Больше или равно | `[PART_WEIGHT] >= 10` |
| `<=` | Меньше или равно | `[PART_WEIGHT] <= 20` |
| `like` | Соответствие маске | `[PART_NAME] like "Трансформатор %"` |
| `not like` | Не соответствует маске | `[PART_NAME] not like "Временный %"` |
| `contains` | Содержит подстроку | `[PART_NAME] contains "Болт"` |
| `not contains` | Не содержит подстроку | `[PART_NAME] not contains "Временный"` |
| `isset` | Параметр существует и не пустой | `isset([CENTROID_POINT_X])` |
| `not isset` | Параметр отсутствует или пустой | `not isset([CENTROID_POINT_X])` |

## Логика сравнения

- Для `>`, `<`, `>=`, `<=` сначала выполняется попытка числового сравнения.
- Числа принимаются с точкой и запятой: `10.5` и `10,5`.
- Если оба значения не удаётся преобразовать в числа, используется строковое сравнение без учёта регистра.
- Для `=` и `<>` используется сравнение без учёта регистра.
- Отсутствующий параметр считается пустой строкой.
- Для числовых сравнений отсутствующий параметр не проходит условие.

## Маски `like`

Поддерживаются два варианта wildcard:

- `%` — как в Model Studio CS;
- `*` — пользовательский алиас для удобства.

Примеры:

```text
[PART_NAME] like "Трансформатор %"
[PART_NAME] like "*Болт*"
```

## Поддержка формул из XML

На текущем этапе поддерживаются простые выражения:

```text
[PART_TYPE]="Арматура" and [BOM_INCLUDE]="1"
[BOM_INCLUDE]<>0 and sort<99
isset([CENTROID_POINT_X])
root.[AEC_WSLICE_VERSION]>2
```

Также поддерживаются простые группы `and` / `or` без вложенных скобок.

## Ограничения текущего этапа

Пока не реализованы:

- вложенные логические выражения со скобками;
- `if(...)` и `case(...)` в фильтрах;
- обращения к реальной иерархии `parent`, `child`, `hierarchy(...)`;
- коллекционные функции `exists`, `all`, `sum`, `avg`, `first`, `last`;
- полноценный визуальный мастер функций.

Префиксы `root.`, `current.`, `object.` на первом этапе нормализуются к текущему набору атрибутов объекта.
